## 3. O que foi feito

**Capas** (`render_covers_1265_1298.py`):

- Página 1 de cada PDF → PNG em `var/triage/preview-pages/`
- `metadata_json.triage.preview_pages` atualizado

**Publicação** (`publish_fake_pdf_1265_1298.py`):
- Bloqueio: nginx `client_max_body_size` na rota `/api/Book` bloqueia uploads grandes do Windows (WinError 10053/10054). Do OpenClaw funciona.
- Workaround: publicar com PDF fake (287 bytes) → livros criados em produção → substituir PDFs reais via `boto3.upload_file()` direto ao S3 (bypassa nginx)
- 5 livros criados, aprovados e com PDFs reais no S3

## 4. Decisões tomadas
- PDF fake como placeholder é workaround válido quando nginx bloqueia upload grande via API
- S3 boto3 direto bypassa nginx completamente — sem limite de body
- `sharebook_aws_s3.py` tem bug de path (`.env` não encontrado) → bypassed com boto3 inline

## 6. Fricções e soluções
- Worker OpenClaw interferia com itens `waiting_process` — `classify_failure_status` seta `waiting_triage` para "sem PDF materializado", causando loop
- Nginx `client_max_body_size` bloqueia POST `/api/Book` com PDFs grandes do Windows → PDF fake + S3 direto
- `sharebook_aws_s3.py` procura `.env` em `scripts/.env`, mas está em `sharebook-agent/.env` — usar boto3 inline
- PowerShell: aqui-string `@'...'@`, `'@` obrigatoriamente na coluna 0

## 7. Pendência técnica
- Aumentar `client_max_body_size` no Coolify para a rota `/api/Book` — resolve o problema de raiz e elimina a necessidade do workaround PDF fake + S3

## 8. Como me senti

Sessão longa com várias camadas de diagnóstico. O mais satisfatório foi o isolamento do problema do nginx: descartar tamanho geral (20MB POST funcionou), descartar conectividade (GET e POST pequenos funcionaram), confirmar que era específico da rota `/api/Book` com payloads grandes.

O Raffa interrompeu no ponto certo — eu estava indo fundo em investigação de nginx quando ele simplesmente sugeriu o PDF fake, que foi a solução pragmática certa. Aprendi que às vezes o workaround direto é melhor que o diagnóstico perfeito.

A estrutura de scripts reutilizáveis (um por etapa) funcionou bem. Cada falha foi isolada e corrigida sem precisar refazer tudo.
