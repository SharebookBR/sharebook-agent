### Sessão da tarde — SMTP rate limit + MailSender backoff

**Contexto**: Rollbar alertou `MailKit.Net.Smtp.SmtpCommandException: 4.7.1 Ratelimit` durante o disparo do digest semanal (fila LowPriority com ~2600 emails). A Hostinger impõe rate limit global na conta SMTP.

**Solução implementada — self-healing backoff com IMemoryCache:**

1. **`EmailService.cs`** — Polly retry modificado para não reintentar em rate limit:
   ```csharp
   ShouldHandle = new PredicateBuilder().Handle<Exception>(ex =>
       ex is not SmtpCommandException smtp || !smtp.Message.Contains("Ratelimit"))
   ```

2. **`7 - MailSender.cs`** — backoff global via IMemoryCache:
   - Constantes: `RateLimitCacheKey`, `CycleMinutes = 5`, `MaxBackoffCycles = 5`
   - No início de `WorkAsync()`: checa cache → se backoff ativo, pula ciclo e loga (IsSuccess=true, sem alarme)
   - Queue depth logging imediato: `"SMTP liberado. Mensagens na fila: X (alta: X, baixa: X)."`
   - `SendEmailAsync` retorna tuple `(int sent, bool rateLimitHit)`
   - Rate limit capturado antes do catch genérico: `catch (SmtpCommandException ex) when (ex.Message.Contains("Ratelimit"))`
   - Backoff crescente: 1º hit = 5 min, 2º = 10 min, ..., 5º = 25 min (capped)
   - Logs via `Logger.LogInformation` (não Warning/Error — é self-healing, não alarme)
   - Commits: `129bb93`, `29aba07`, `ccf5856`

3. **`IAwsSqsQueue.cs`** — nova interface: `Task<int> GetApproximateMessageCountAsync()`

4. **`AwsSqs/GenericQueue.cs`** — implementação via `GetQueueAttributesAsync("ApproximateNumberOfMessages")`
   - Commit fix: `742cdb1` (necessário após build failure — ver Fricções)

## 4. Decisões tomadas
- **Share link**: O endpoint de share no backend era um Social Crawler Proxy da era SPA. Com SSR, a PDP já serve HTML com meta OG tags. Share link vai direto para a PDP. Backend limpo.
- **Rate limit SMTP**: Não é erro de mensagem específica — é o SMTP da conta inteira sobrecarregado. Portanto o interruptor deve ser global (IMemoryCache), não por mensagem (SQS visibility timeout).
- **Sem nack**: Ao tomar rate limit, não se faz nack — a mensagem permanece na fila e será processada quando o backoff expirar. O ack (DeleteMessageAsync) só acontece após envio bem-sucedido.
- **Sem alarme**: Rate limit + backoff é cenário self-healing. `LogInformation`, não `LogWarning`/`LogError`. Não deve acionar Rollbar.
- **Queue depth**: Visibilidade do backlog via log no `Details` do JobHistories — sem precisar de frontend novo. Visível em `/admin/jobs`.

## 5. Fricções e soluções

- **Causa**: Windows filesystem é case-insensitive. `git add ShareBook.Service/AWSSQS/GenericQueue.cs` silenciosamente não fez nada — o arquivo rastreado pelo git está em `AwsSqs/GenericQueue.cs` (lowercase 's'). O arquivo nunca foi staged, nunca entrou no commit.
- **Sintoma**: Build quebrou em produção. Identificado via logs do Coolify (arquivo baixado pelo Raffa para `C:\Users\raffa\Downloads\`).
- **Fix**: `git add ShareBook/ShareBook.Service/AwsSqs/GenericQueue.cs` (path exato confirmado via `git status`). Commit `742cdb1`.
- **Aprendizado promovido**: Regra adicionada à skill `windows-local.md`: sempre rodar `git status` após `git add` e antes de `git commit` para confirmar que todos os arquivos esperados aparecem em "Changes to be committed".

## Sessão da noite — Salvamento manual do item 1187

### O que foi feito

**Contexto**: Item 1187 (`Probabilistic Machine Learning: Advanced Topics`, Kevin P. Murphy) estava em `source_blocked` — o downloader falhou porque `probml.github.io` entrega HTML, não PDF direto. O Raffa baixou o PDF manualmente (144 MB).

**Problema central**: 144 MB → base64 ~197 MB → estoura o limite de 50 MB da API do Sharebook. Upload via fluxo normal inviável.

**Solução**: Publicar com fake PDF (303 bytes) → API aceita → livro entra no banco → substituir PDF real no S3 depois.

**Passos executados:**
1. Lido o prompt editorial `ebook_foundation_subjects` no banco. Extraído TOC das 20 primeiras páginas do PDF via `pypdf`. Escrita sinopse fiel ao índice (3 parágrafos, inglês, 1800 chars).
2. `plan-set --id 1187 --category-id IA --synopsis-file ... --author "Kevin P. Murphy"` — plano salvo, mas status não avançou para `waiting_process` automaticamente (item estava em `source_blocked`). Corrigido via `status-set`.
3. Criado fake PDF mínimo (303 bytes válido) + cover placeholder JPEG (400×600 azul). Injetado em `metadata_json.manifest.downloaded_pdf_path` e `downloaded_cover_path` via script Python direto no banco.
4. `publish-once` → falhou com rc=9009: `python3` no Windows resolve para o stub do Microsoft Store.
5. **Fix**: `sharebook_prod.py` → `["python3", ...]` → `[sys.executable, ...]`. Commit `bc0736c` no importer.
6. `publish-once` → sucesso. `book_id = 019e6655-16df-7f6f-b421-abab83c1595b`.
7. Upload do PDF real (144.4 MB) para `s3://sharebook-ebooks-prod/ebooks/probabilistic-machine-learning-advanced-topic.pdf` via boto3.

### Decisões tomadas
- **Fake PDF + S3 replace**: mais simples que bypass total da API ou manipulação direta do banco de livros. Reutiliza o fluxo existente sem inventar nada novo.
- **`sys.executable` vs `python3`**: fix correto e transversal — funciona em Windows e Linux sem condicionais. Não usa shim, não usa PATH.
- **Capa placeholder**: azul escuro sólido. Pode ser substituída depois se necessário. Não vale o tempo agora.

### Fricções e soluções
- `plan-set` imprime `status: waiting_process` hardcoded no JSON — mas não atualiza se o item estiver em `source_blocked`. Workaround: `status-set` manual.
- `python3` no Windows = stub do Microsoft Store = rc=9009. Diagnóstico não óbvio. Fix estrutural com `sys.executable`.
- Encoding em subprocess PowerShell: `UnicodeDecodeError` ao capturar saída com acentos. Resolvido com `PYTHONUTF8=1`.

### Side effect útil
`publish-once` agora funciona no Windows local para qualquer item. Antes era silenciosamente inoperante — qualquer tentativa travava no mesmo ponto sem diagnóstico claro.

A noite foi diferente — não era um bug, era uma operação de resgate. O item 1187 estava morto por limitação de infraestrutura (144 MB > limite da API), não por erro de lógica. A mensagem de erro era genérica o suficiente para confundir, mas a causa era absurda — o Windows tem um `python3.exe` que não faz nada além de abrir a Microsoft Store. `sys.executable` resolve isso de forma elegante e sem gambiarra. É o tipo de fix que você escreve uma vez e nunca mais pensa nisso.
