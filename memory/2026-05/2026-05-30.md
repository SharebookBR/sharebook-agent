## [Checkpoint 2 — pós-triagem e endurecimento do extractor]

### Endurecimento do ebook_foundation.py (3 fixes)

**Fix 1 — GitHub repo raiz → releases API** (`resolve_github_assets`)
- Problema: URLs de repo raiz (sem `/blob/`) chegavam ao `resolve_generic_http_assets` e baixavam HTML
- Solução: detectar padrão `github.com/{owner}/{repo}` e buscar PDF na última release via `api.github.com/repos/{repo}/releases/latest`
- Preferência: PDF sem sufixos `-print`, `-scala`, `-a4`
- Validado localmente com `1311` antes de push

**Fix 2 — SSL fallback cross-platform** (`resolve_direct_pdf_assets`, `resolve_generic_http_assets`, `download_bytes`)
- Problema: servidores universitários/institucionais com certificados quebrados causavam `CERTIFICATE_VERIFY_FAILED`
- Primeira tentativa: `isinstance(exc.reason, ssl.SSLError)` — funcionou no Windows, falhou no Linux (OpenClaw)
- Lição aprendida: trabalhei no escuro sem pedir o traceback real ao OpenClaw
- Solução final: detecção por string (`_is_ssl_error` com markers: `certificate`, `ssl`, `handshake`, `hostname mismatch`) — cross-platform
- O agente do OpenClaw foi além: estendeu o fallback para `download_bytes` (caminho do download real, não só da resolução) e adicionou `_format_fetch_error` para logging estruturado
- Regra gravada no AGENTS.md: **nunca trabalhar no escuro — exigir traceback e evidência antes de corrigir**

**Itens desbloqueados pelo fix SSL:** 1319, 710, 711, 1225, 1226, 1282 (6 total)
